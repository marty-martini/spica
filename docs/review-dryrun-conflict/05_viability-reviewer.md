# viability-reviewer 生出力

## Assumptions

- 「直近10サイクル分で平均2,000トークン」がGenerator/Checker各1回分か、両者合算か、リトライ前提を含むかの内訳が記載されていないため不明。
- 想定発火頻度「1日あたり平均3回」がユーザー1人あたりか、システム全体(全ユーザー合算)かが記載されていないため不明。
- 使用モデル(価格帯)、入出力トークンの内訳比率が記載されていないため、金額換算ができない。
- 「実装3日・テスト1日」の見積もりが何人日か(1人か複数人か)、また保守運用フェーズの工数(監視・アラート対応・障害時対応)が記載されていないため不明。
- 自動リトライの上限回数・リトライ発生率の実測または推定値が記載されていないため不明。

## Findings

- 「自動リトライ発生時は追加でGenerator/Checkerサイクル分のAPIコストが発生するが、発生頻度が低いため試算は割愛する」と書かれている。発生頻度が低いという根拠(実測値・過去データ・類似機能の統計等)は本文中に示されていない。
- コスト試算の対象が「平均2,000トークン×1日3回」の通常フローのみであり、リトライ分・失敗時の再送分・将来的な発火頻度増加(ユーザー数増加時)のシナリオが試算に含まれていない。
- スケジュールは「実装3日・テスト1日」の合計4日のみが記載されており、リリース後の運用・監視・アラート誤発火対応などの保守工数への言及がない。
- 「今週末のボルトで着手する」と書かれているが、このボルトが何を指す単位か(スプリント/開発イベント)の定義や、既存の他タスクとのトレードオフ(このボルトで他に何を積む予定だったか)が本文に記載されていない。

## Risks

**[最重要] リトライコストが「試算割愛」とされているが、リトライは失敗時に発生するものであり、失敗率が高い局面(API障害・ネットワーク不調・Checker否認の連鎖など)ほどリトライ回数が跳ね上がる性質を持つ。ランニングコストの試算から最も変動しやすい項目を除外した状態でスケジュール(4日)に着手判断を下しており、実運用開始後にコスト超過が発覚するリスクがある。**

- 発火頻度「1日3回」が固定値として扱われているが、対象ユーザー数やループ数のスケール想定が記載されておらず、将来的な利用拡大時にコストが線形以上に増える可能性を検証する材料がない。
- 実装・テストの4日見積もりに対し、保守負荷(アラート運用・誤検知対応・リトライロジックの障害対応)の工数が試算に含まれておらず、初期実装コストのみで「割に合うか」を判断している。

## Recommendation

- 「発生頻度が低い」の根拠となる数値(想定リトライ率、例:5%など)を最低限1行加え、リトライ分を含めた上限コスト(ワーストケース試算)を併記する。試算を作り直す必要はなく、通常時試算に「リトライ率X%時の追加コスト」を1行足すだけで足りる。
- スケジュールに保守フェーズの見積もり(例:月あたり監視0.5人日など)を1行追加し、初期4日だけでなく運用込みでの工数感を明示する。
