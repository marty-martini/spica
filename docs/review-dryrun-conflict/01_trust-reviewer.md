# trust-reviewer 生出力

## Assumptions

- construction-loopのGenerator/Checkerサイクルが実際に何を実行するか(コード生成のみか、コード実行・外部API呼び出しを伴うか)は本抜粋に記載がなく不明。
- 「該当ループ」が誰の管理下にあるプロジェクト/リポジトリに対して自動リトライを行うのか、対象範囲の記載がなく不明。
- 自動リトライ実行者(実行権限・使用する認証情報が開発者本人のものか、システム共通のサービスアカウントか)についての記載がなく不明。
- 自動リトライ2回の間隔・タイムアウト・同時実行制御についての記載がなく不明。
- Slack通知の宛先が誰か(開発者本人のみか、チャンネル全体か)、および通知に含まれる情報の範囲についての記載がなく不明。

## Findings

- 機能概要3.に「開発者の確認を待たずに該当ループを自動リトライする」と明記されており、実行前の人間による承認ステップが存在しない。
- 機能概要5.に「実行後の事後報告のみで、実行前の承認フローはない」と明記されており、承認フローの欠如が設計として意図的に選択されている。
- 技術メモに「自動リトライの一時停止・無効化スイッチ(緊急停止)は今回のスコープには含めない(将来検討)」と記載されており、稼働中に問題が発生した場合でも人間が即座に介入する手段が本バージョンには存在しない。
- 機能概要4.では自動リトライの上限が「1インテントにつき最大2回」と定義されているが、リトライ発生の頻度・時間内での総リトライ数(複数インテントが同時多発した場合の全体量)を制御する仕組みの記載がない。

## Risks

**[最重要] 実行前承認なし・緊急停止スイッチなしで自律的にコード生成/実行サイクルを繰り返す設計になっている。** Generator/Checkerサイクルの中身(コード生成・実行の範囲)が不明な状態で、閾値割れという自動判定のみをトリガーに、人間の承認なく最大2回まで自動でループを再実行する。これは事後報告のみのAI Safety上のヒューマン・イン・ザ・ループの欠如であり、意図しない/予期しないリトライの連鎖が発生した場合に、緊急停止手段(スコープ外)がないため、開発者はSlack通知を読むまで停止も介入もできない。誤ったスコアリングロジックや外部要因によるスコア低下が続いた場合、無害化されない自動処理が繰り返される経路が残る。

- 事後報告のみの通知設計(機能概要5.)は、承認なき自動処理という点でResponsible AIの観点から説明責任(なぜ実行されたか、実行前に止める手段があったか)が弱い。開発者が「実行されたことを知る」だけで「実行を止める・許可する」機会がない。
- リトライ上限到達時のSlack通知文言(機能概要4.)は結果の告知のみであり、リトライ中に何が変更された/生成されたかのログ・監査証跡の記載が本抜粋にない。自動処理の内容を後から検証する手段が明記されていないため、誤動作や誤ったコード生成が発生した際の追跡可能性が担保されているか不明。

(参考・担当外)自動リトライが繰り返し実行された場合のリソース消費・可用性影響についてはEngineering Reviewerの確認を推奨。

## Recommendation

- 緊急停止スイッチを「将来検討」から本バージョンの必須スコープに引き上げるか、それが難しい場合は最低限「Slack通知に取り消し/停止を促すアクション(例: リアクションやコマンドでの停止依頼)」を暫定的に追加し、稼働中に人間が介入できる経路を1つ確保する。
- 自動リトライ実行時に、何が生成・変更されたかを追跡できる最小限の監査ログ(実行ログへのリンクなど)をSlack通知に含める。
