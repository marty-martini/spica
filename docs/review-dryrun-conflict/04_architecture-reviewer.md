# architecture-reviewer 生出力

## Assumptions

- 「直近2回のポーリングサイクル」の間隔・タイムアウト定義がドラフト内に記載されていないため不明。
- 「改善しない」の判定基準(スコアがどの程度回復すれば「改善」とみなすか)が記載されていないため不明。
- 自動リトライ中に開発者が手動で介入(修正・キャンセル)した場合の扱いがドラフト内に記載されていないため不明。
- 「1インテントにつき最大2回まで」のカウンタのリセット条件(いつゼロに戻るか、恒久的な累積か)が記載されていないため不明。
- construction-loopのGenerator/Checkerサイクルの既存インターフェース(呼び出しトリガーが人間の確認のみを前提としていたか)がこのドラフトからは不明。

## Findings

- 技術メモに「自動リトライの一時停止・無効化スイッチ(緊急停止)は今回のスコープには含めない(将来検討)」と明記されている。緊急停止機構を持たない自動実行系を先に組み込み、後から挿入する設計になっている。
- 機能概要 3. に「開発者の確認を待たずに該当ループを自動リトライする」とあり、人間承認ゲートをバイパスする分岐が新設される。既存のconstruction-loopが人間承認を前提に設計されていたか、疎結合な形でこの分岐を挿し込めるかはドラフトに記述がない。
- 機能概要 4. に「自動リトライは1インテントにつき最大2回まで」とあるが、リトライ回数のカウント主体・保存場所についての記述がない。「該当ループを自動リトライする」(3.)という記述の粒度が「インテント単位」(4.)とずれており、ループとインテントの対応関係(1インテント=1ループなのか、複数ループなのか)が定義されていない。
- 技術メモに「リトライ中に生成される差分・ログは、通常のループと同様に `.loop/` 配下に記録される」とあるが、通常の(人間承認を経た)ループの記録と自動リトライによる記録を区別するフィールド・フラグの記述がない。将来、自動リトライ由来かどうかを判別する要件(監査・分析)が出た場合、`.loop/` のスキーマ変更が必要になる可能性がある。
- (参考・担当外)自動リトライ中の外部呼び出しコストや無限リトライ的な挙動によるリソース消費については Engineering Reviewer / Viability Reviewer の確認を推奨。

## Risks

**[最重要] 緊急停止スイッチを「将来検討」としてスコープ外にしたまま、人間承認バイパスの自動実行ロジックを組み込む設計になっている。** 自動リトライが本番導入された後で緊急停止機構を追加しようとすると、既にリトライトリガー・カウンタ・通知ロジックが「常時有効」を前提に実装されているため、スイッチによる分岐点を後付けで挿入する箇所(トリガー条件のどこにチェックを挟むか)の設計変更が必要になる。安全装置は初期設計の分岐点に組み込むかどうかで改修コストが大きく変わる典型例であり、後回しにするほど曲げにくくなる。

- リトライ回数カウンタの保存主体・粒度(インテント単位かループ単位か)が未定義のまま実装が進むと、後で粒度を変更する際に既存データの移行・整合性の取り直しが必要になる。
- `.loop/` の記録に自動リトライ由来かどうかを示す区別がない場合、監査要件やUI表示要件が後から追加された際にログスキーマの後方互換性のない変更が必要になる可能性がある。

## Recommendation

- 緊急停止スイッチを「将来検討」から外し、今回のリリースでは最小実装(例: 設定ファイルの真偽値1つ、リトライトリガー分岐の先頭でチェックするだけ)として組み込む。トリガー分岐点さえ用意しておけば、UIや運用フローは後から拡張できる。
- リトライ回数カウンタの保存単位(インテントID/ループID/実行ID)を明記し、`.loop/` ログに `retry_triggered_by: auto | manual` のような区別フィールドを最初から含めておく。
