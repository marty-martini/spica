# engineering-reviewer 生出力

## Assumptions

- 「ポーリングサイクル」の実行間隔(何秒/分ごとにファイル監視結果を評価するか)が仕様上明記されていないため、「2回連続」がどの時間スケールを指すか不明。
- ファイル監視(watch)の実装方式(inotify/fswatch的な即時検知か、定期ポーリングか)が「ファイル監視で検知」としか書かれておらず、機能概要1と受入基準・技術メモの「ポーリングサイクル」という語との整合性が不明。
- 自動リトライ処理自体(construction-loop再実行プロセス)の稼働状態を監視するプロセス管理・ヘルスチェックの仕組みについて記載がなく不明。
- Slack Webhook送信やLLM要約呼び出しが失敗した場合のリトライ・フォールバック仕様が記載されておらず不明。
- 本機能自体(監視デーモン/ジョブ)のデプロイ方法・稼働環境(常駐プロセスかCIジョブか等)が記載されておらず不明。

## Findings

- 機能概要3・受入基準に「2回連続で閾値を下回った場合、開発者への確認プロンプトなしに自動リトライが起動すること」と明記されており、実行前承認フローがないまま本番のconstruction-loopを自動的に再実行する設計になっている。
- 技術メモに「自動リトライの一時停止・無効化スイッチ(緊急停止)は今回のスコープには含めない(将来検討)」と明記されている。自律的にループを再実行する機能でありながら、稼働中に異常を検知した際の停止手段が存在しない。
- 受入基準に「健康度スコアが閾値を下回ってから60秒以内にSlack投稿が成功すること」とあるが、機能概要2では「直近ログをマスキング処理した上でLLMに要約させ」てから投稿するとされており、LLM呼び出しの遅延・失敗時にこの60秒SLAをどう担保するかの記載がない。
- 機能概要3・4・5、および受入基準に自動リトライ発生・上限到達をSlackに事後通知する旨は書かれているが、リトライの実行ログ・成功/失敗・所要時間といった技術的テレメトリ(メトリクス化・ダッシュボード化)についての記載が一切ない。Slack通知文言のみが唯一の観測手段として設計されている。
- 技術メモに「自動リトライは、該当インテントのconstruction-loopを最初のステップ(Generator呼び出し)から再実行する」とあるが、リトライ中に元の(詰まっていた)ループプロセスが停止しているかどうか、二重起動を防ぐ排他制御についての記載がない。
- 「リトライ中に生成される差分・ログは、通常のループと同様に `.loop/` 配下に記録される」とあるが、この記録がスコア算出用の「直近10サイクル」の集計対象に含まれるかどうかの記載がなく、自動リトライがトリガーとなって誤って追加のアラート/リトライを誘発する可能性(フィードバックループ)について検討された形跡がない。
- コスト試算に「自動リトライ発生時は追加でGenerator/Checkerサイクル分のAPIコストが発生するが、発生頻度が低いため試算は割愛する」とあるが、これはコスト観点であり実行時負荷(同時実行数、監視プロセスとリトライプロセスのリソース競合)については触れられていない。
- スケジュールに「実装3日・テスト1日を見込み、今週末のボルトで着手する」とあるのみで、デプロイ手順・ロールバック手順(本機能自体に不具合があった場合にどう無効化するか)についての記載がない。

## Risks

**[最重要] 自動リトライの緊急停止手段が「今回のスコープ外」と明記されている一方、実行前承認なしに本番ループを自動再実行する設計であり、意図しない暴走(誤検知による無限リトライ的挙動、リトライ自体が新たな差し戻しを生みスコアがさらに悪化するフィードバックループ等)が発生した場合、開発者が稼働中にそれを止める手段が仕様上存在しない。夜間・育児中など「開発者が対応できない時間帯」を主眼に置いた機能であるため、まさにその時間帯に停止できない状態で暴走するリスクが最も高い。**

- Slack投稿60秒SLAの達成可否がLLM要約呼び出しの遅延・失敗に依存する構造だが、LLM呼び出し失敗時のフォールバック(要約なし投稿、リトライ、タイムアウト時の代替経路)が定義されておらず、SLA未達が常態化する可能性がある。
- 自動リトライ実行と、既存の(詰まっている)ループプロセスとの排他制御が未定義であり、二重実行によるリソース競合や不整合な差分生成が起きるリスクがある。
- 監視・リトライ機構自体の稼働状態(プロセスダウン、ファイル監視の取りこぼし)を検知するテレメトリがなく、「アラート機構自体が止まっていることに誰も気づけない」というサイレント障害のリスクがある。

## Recommendation

- 緊急停止スイッチは「将来検討」ではなく本v0.3のスコープに含める。最低限、環境変数やフラグファイル1つでリトライ機能のみを即座に無効化できる仕組み(既存のアラート通知は維持)を先に用意する。
- 「ポーリングサイクル」の間隔を数値で明記し、ファイル監視方式(イベント駆動かポーリングか)と用語を統一する。
- LLM要約・Slack Webhook呼び出しにタイムアウトとフォールバック(要約失敗時は生ログの先頭N行をそのまま投稿する等)を定義し、60秒SLAの達成手段を明確にする。
- 自動リトライ実行前に、対象ループが既に稼働中でないことを確認する排他制御(ロックファイル等)を最小限追加する。
- 自動リトライの発生・成否をSlack通知だけでなく構造化ログ(件数・所要時間・成否)として `.loop/` 配下等に残し、後から集計・傾向分析できるようにする。
