# verification-measurement-reviewer 生出力

## Assumptions

- 「Checker差し戻し率」「サイクル所要時間」の算出元となる `.loop/state.json` のスキーマ・各サイクルの開始/終了タイムスタンプの記録有無が抜粋に記載されていないため不明。
- 「発生時刻」がスコア算出時刻・閾値割れ検知時刻・通知投稿時刻のいずれを指すか記載がないため不明。
- Slack投稿失敗時(Slack API障害・ネットワーク断など)の扱いが受入基準に記載がないため不明。
- 自動リトライの「失敗」の判定基準(何をもって失敗とするか)が記載されていないため不明。
- 閾値70点が「暫定値」とされているが、暫定から確定値へ移行する基準・プロセスが記載されていないため不明。

## Findings

- 受入基準に「閾値を下回ってから60秒以内にSlack投稿が成功すること」とあるが、「閾値を下回った」時刻の算出元・検知トリガー(ポーリング間隔かイベント駆動か)が仕様上定義されていない。60秒SLAを機械判定するには基準時刻の定義が不可欠だが、その定義が本抜粋に存在しない。
- 通知本文の必須4項目(スコア値・閾値・差し戻し率・発生時刻)は列挙されており、Checkerが文字列パターンマッチで機械判定可能な形になっている。
- 「2回連続で閾値を下回った場合」「自動リトライが2回失敗した場合」という条件は数値化されており判定可能だが、"連続"の定義(直近2回のスコア算出サイクルが連続していることの判定基準、途中でスコアが閾値を上回った場合のカウントリセット有無)が記載されていない。
- 健康度スコアの加重平均式(差し戻し率60%・所要時間40%)は数式として定義されているが、「平均サイクル所要時間(過去7日平均との比較)」をどう0〜100のスコアに正規化するかの変換式が記載されておらず、この部分は現状のままでは機械的に再現・検証できない。
- 閾値70点について「暫定値であることを仕様上明記する」とあるが、それ自体が受入基準やKPIとして検証可能な形になっておらず、単なる注記に留まっている。閾値の妥当性を事後検証する指標(例:誤検知率・見逃し率)が定義されていない。
- 「自動リトライが起動すること」「リトライを停止し、その旨がSlack通知に含まれること」は機能の作動有無としては機械判定可能だが、自動リトライという副作用の大きい動作(開発者の確認プロンプトなしに起動)に対して、リトライ発火・停止のログ記録(監査可能な形での記録)が受入基準に含まれていない。
- KPI・成功指標(この機能が「成功している」とどう判断するか、例:誤通知率、通知の実効性、リトライによる自己修復成功率など)がこの抜粋には一切記載されていない。受入基準は個別動作の合否のみで、機能全体の成功指標が欠落している。

## Risks

**[最重要] 健康度スコアの正規化式が未定義であり、Checkerが機械的に再現・検証できない**
差し戻し率は「直近10サイクル中の割合」で0〜100%に自然に落ちるが、「平均サイクル所要時間(過去7日平均との比較)」は単位も正規化方法(比率化・上限キャップの有無など)も記載がなく、0〜100のスコアに変換する式が仕様上存在しない。この状態では同じ入力データからでも実装ごとに異なるスコアが算出され得り、「閾値を下回ったか」という全受入基準の前提そのものが検証不能になる。

- 自動リトライという不可逆・副作用のある挙動(開発者確認なしに起動)の発火条件・停止条件・実施結果を追跡する監査ログ要件が受入基準に含まれておらず、誤発火・過剰リトライが発生しても事後に検証できない。
- 「発生時刻」の定義が曖昧なため、60秒SLA達成の判定基準点が実装依存になり、Checkerごとに合否が割れる可能性がある。
- 機能全体としての成功/失敗を測るKPI(例:誤通知率、リトライ成功率、閾値の妥当性を測る事後指標)が定義されておらず、個々の動作テストにパスしても機能が「意図通り機能しているか」を判断する手段がない。

## Recommendation

- 「平均サイクル所要時間」から0〜100スコアへの正規化式を明記する(例:過去7日平均比の何%超過で何点減点、上限・下限の定義)。
- 「発生時刻」の定義(スコア算出時刻/閾値割れ検知時刻のいずれか)を一文で明記する。
- 自動リトライの発火・停止イベントを `.loop/state.json` 等に記録する要件を受入基準に1行追加する(事後監査のため)。
- 「連続」判定のロジック(リセット条件含む)を1文で補足する。
- 機能全体のKPI(例:週次の誤通知率、閾値見直しの判断材料となる事後集計指標)を最低1つ追加する。
