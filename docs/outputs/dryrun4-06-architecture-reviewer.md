# architecture-reviewer 生出力

## Assumptions

- 「Checker差し戻し率」「平均サイクル所要時間」の具体的な計算式(分子・分母の定義、正規化方法)が成果物に記載されていないため不明。
- 重み60/40という数値の決定根拠(実験・過去データに基づくものか、暫定的な初期値か)が記載されていないため不明。
- `.loop/config.json` のスキーマ(閾値以外にどの項目を外出しする想定か)が記載されていないため不明。
- 直近10サイクルの「10」という件数の決定根拠、および将来調整時の変更手段(config化するのか、コード定数のままか)が記載されていないため不明。
- スコア算出ロジック(加重平均の重み60/40)自体が `.loop/config.json` に含まれるのか、コード内に固定されるのかが記載されていないため不明。

## Findings

- 「技術メモ」に「閾値(70点)は `.loop/config.json` に外出しし、コード変更なしで調整可能にする」と明記されており、閾値単体については外部化する意図が構造として示されている。
- 一方で「健康度スコアの算出方法」には加重平均の重み(60%/40%)が本文中に直接記述されており、これが `.loop/config.json` の対象に含まれるか、コード定数として扱われるかの記載がない。閾値のみを外部化する設計思想と、重みの扱いに一貫性があるかどうかが記述からは判断できない。
- 「運用開始後2週間のアラート発火実績(誤検知件数・見逃し件数)を記録し、閾値の妥当性を見直す」とあり、閾値については見直しサイクルが明記されているが、重み(60/40)についての見直し・再調整のプロセスは記述されていない。閾値が「暫定値」と明記されているのに対し、重みの暫定性については言及がない。
- 「LLMに渡すログは直近10サイクル分に固定し、上限を超えない」と記載されており、上限が「10」という数値でハードコードされる前提が読み取れる。この値が `.loop/config.json` に含まれるかどうかは技術メモ内で明言されていない(閾値のみが名指しで外部化対象とされている)。
- 「健康度スコアの算出方法」のセクションでは、差し戻し率と所要時間という2つの異質な指標(比率 vs 時間比較)を単一の加重平均式に統合する旨が記載されているが、両者をどう正規化して0〜100スケールに合成するかの算出式は記載されていない。将来的に指標を追加・変更する際の拡張ポイント(例:第3の指標を足す場合の重み再配分方法)についても記述がない。

## Risks

**[最重要] 加重平均の重み(60/40)とログ件数上限(10)が、閾値と異なり外部化対象として明記されていない**
技術メモは「閾値(70点)は `.loop/config.json` に外出し」と閾値のみを名指ししており、同じ「後で調整するかもしれない値」であるはずの重み60/40やログ件数10がコード内に埋め込まれる可能性がある。閾値は「運用開始後2週間の実績を見て妥当性を見直す」と明記されているのに対し、重みや件数の見直しトリガーが定義されていないため、これらの値を変更する際に閾値と同じ「コード変更なしで調整」という運用ができない構造になるおそれがある。

- 差し戻し率と所要時間という異質な指標を合成する正規化ロジックの算出式が本文にないため、将来「第3の指標を追加する」「重み配分を変える」といった変更が、どの程度コードに手を入れずに行えるかが仕様上判断できない。

## Recommendation

- `.loop/config.json` の対象範囲を明記し、少なくとも「重み(60/40)」と「ログ件数上限(10)」を閾値と同様に外部化対象に含めるか、あるいは意図的にコード固定とするのかを一文で仕様に明記する。
- 加重平均の正規化方法(各指標を0〜100にどう変換して合成するか)を仕様に1〜2行追記し、将来指標を追加・変更する際の変更単位を明確にする。
