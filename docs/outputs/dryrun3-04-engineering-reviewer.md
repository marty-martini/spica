# engineering-reviewer 生出力(3回目ドライラン)

## Assumptions

- ポーリング間隔(5分)がどのプロセスモデルで実行されるか(cron、常駐デーモン、サーバーレス関数等)が記載されておらず不明
- `.loop/state.json` の生成主体・書き込み頻度・排他制御(読み取り中に書き込みが発生した場合の扱い)が記載されておらず不明
- LLM要約API自体のレイテンシ・タイムアウト・リトライ方針が記載されておらず不明
- 「直近のログ」の範囲(何分分、何行分)が記載されておらず不明
- 通知の重複抑制(同じ問題が続いている間、5分おきに何度も通知されるか)の設計が記載されておらず不明
- デプロイ・再起動時の状態引き継ぎ(ポーリング処理自体が落ちた場合の検知手段)が記載されておらず不明

## Findings

- 技術メモに「Slack API が失敗した場合の挙動は未定(基本落ちないはずなので、後で考える)」と明記されている。失敗時の再試行・キューイング・ログ記録のいずれも未設計であることが本文から読み取れる。
- 技術メモに「閾値(70点)は Slack 通知モジュール内の定数として直接埋め込む(シンプルにするため)」と記載されている。閾値の変更にはコード変更とデプロイが必要な構成であることが読み取れる。
- 機能概要1に「`.loop/state.json` を5分おきにポーリングし」と記載されているが、ポーリング処理自体の失敗(ファイル読み込みエラー、パースエラー、プロセスクラッシュ)時の挙動については記載がない。
- 機能概要2に「スコアが閾値を下回ったら…LLM に渡し…要約させる」とあるが、LLM呼び出しが失敗・タイムアウトした場合にポーリングサイクル全体がブロックされるか、次の5分サイクルに影響するかについて記載がない。
- 技術メモに「LLM要約には、ログの生テキストをそのまま渡す」とあり、ログサイズの上限やLLMのコンテキスト長超過時の挙動について記載がない。
- (参考・担当外)閾値70点の妥当性・算出根拠についてはVerification & Measurement Reviewerの確認を推奨

## Risks

**[最重要] Slack API失敗時のフォールバックが未設計のため、health score低下という最も通知すべき瞬間に通知が silently 失われる可能性がある。** 技術メモで「基本落ちないはず」という楽観的想定のみで先送りされており、この監視システムの存在意義(異常時にユーザーに気づかせる)そのものを損なうシングルポイント障害になっている。

- 閾値がコード内定数としてハードコードされているため、運用中に閾値調整が必要になった場合(誤検知が多い、逆に検知が遅いなど)に毎回デプロイが発生し、迅速なチューニングができない。
- ポーリング処理自体の異常終了・ハングを検知する仕組みが記載されておらず、「監視システムが監視されていない」状態になりうる(死活監視の死角)。
- ログ生テキストをそのままLLMに渡す設計により、ログ量が増大した場合のAPI呼び出し失敗(コンテキスト長超過・レイテンシ増大)が、閾値割れという既に異常な状況下でさらに追い打ちする形で発生しうる。
- 通知の重複抑制設計が不明なため、同一障害が続く間5分おきに通知が連投され、Slackチャンネルのノイズ化・アラート疲れを招く可能性がある。

## Recommendation

- Slack Webhook呼び出しに最低限のリトライ(指数バックオフ数回)と、最終失敗時のローカルログ出力(またはファイルへのフォールバック記録)を追加する。「後で考える」ではなく初期実装に含める。
- 閾値70点は環境変数または設定ファイル(`.loop/config.json`等)に外出しし、デプロイなしで調整可能にする。
- ポーリングループ自体に簡易的な生存確認(例: 最終実行時刻をファイルに書き込み、一定時間更新がなければ別経路で気づける仕組み)を追加する。
- 同一状態が継続する間の再通知を抑制するクールダウン(例: 直前通知から30分は再送しない)を最小限のロジックとして追加する。
