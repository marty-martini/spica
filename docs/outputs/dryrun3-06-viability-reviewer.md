# viability-reviewer 生出力(3回目ドライラン)

## Assumptions

- 使用する LLM モデル名・API 単価が成果物に記載されていないため不明
- 「直近のログ内容」の想定サイズ(行数・文字数・トークン数)が記載されていないため不明
- 監視対象(.loop/state.json)の数・プロセス数が単一か複数かの記載がないため不明
- 閾値を下回った状態がどの程度の頻度・継続時間で発生しうるかの想定が記載されていないため不明
- インフラ(ポーリングを実行する主体・実行環境)の費用が記載されていないため不明
- 月間の運用継続期間や利用者数など、コストを積算する母数が記載されていないため不明

## Findings

- 「LLM 要約のAPIコストは、そんなに大した額にはならないはず」と書かれているのみで、具体的な金額・トークン数・呼び出し回数による試算は示されていない。
- 「5分おきのポーリングも軽い処理なので問題ないと思う」とあるが、ポーリング自体の軽さと、閾値割れ時に発生するLLM呼び出しのコストは別物であり、この一文はポーリング処理コストにしか言及しておらず、LLM呼び出し頻度・累積コストには触れていない。
- 「スコアが閾値を下回ったら、直近のログ内容をそのまま LLM に渡し要約させる」と書かれており、ログを事前に切り詰めず「そのまま」渡す設計になっている。ログ量の上限が定義されていないため、1回あたりの入力トークン数が見積もれない。
- 「Slack Webhook 経由で投稿する」という後段はあるが、閾値割れが継続した場合に要約→投稿が何度繰り返されるか(連続ポーリングごとに毎回LLM呼び出しが走るのか、クールダウンがあるのか)についての記載がない。

## Risks

**[最重要] LLM呼び出しコストが「そんなに大した額にはならないはず」という感覚値のみで語られ、定量的な試算が一切なされていない。** 特に「そのまま渡す」ログが無制限の場合、閾値割れが続く状況(=まさに障害調査で頻発しうる状況)で5分おきに大きめのログを要約し続けると、平常時の想定より大幅にコストが跳ね上がる可能性がある。良好時と悪化時でコスト構造が非対称であることが見落とされている。

- 閾値割れの継続時に呼び出し頻度を抑制する仕組み(クールダウン・重複抑制)の記載がなく、最悪ケース(5分おきに毎回LLM要約)でのコスト上限が未定義。これは工数以前に「上限のないランニングコスト」という継続可否に直結するリスク。
- ログサイズの上限が未定義なため、1回あたりのコストが将来的なログ肥大化(依存サービス増加など)に対して線形以上に増える可能性があり、保守フェーズでのコスト再点検が計画されていない。

## Recommendation

- 「そのまま渡す」を「直近N行 or Mトークンに切り詰めて渡す」に変更し、1回あたりの入力トークン数の上限を明文化する。
- 閾値割れ継続時のクールダウン(例: 前回要約から最低X分は再要約しない)を入れ、最悪ケースの月間呼び出し回数に上限を設ける。
- 「大した額にはならないはず」を、想定ログ行数×トークン単価×想定呼び出し回数/月 の簡易試算(数式1つで良い)に置き換え、月額の概算レンジを一度出す。
