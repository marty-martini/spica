# verification-measurement-reviewer 生出力

## Assumptions

- 「誤検知件数」「見逃し件数」の判定基準(何をもって誤検知/見逃しとするか)が成果物に記載されていないため、運用者がどう判定するのか不明。
- 「実際の放置時間」の計測方法(いつを「詰まりの発生時刻」、いつを「気づいた時刻」とするか)が記載されていないため、90分→15分という目標値の測定手段は不明。
- 2週間後の閾値見直しにおける具体的な判断基準(何件以上の誤検知/見逃しがあれば閾値を変更するか)が記載されていないため不明。

## Findings

- 受入基準は「60秒以内にSlack投稿」「本文に4項目含む(スキーマ検証)」「機密パターンがテストケースで検出されないこと」「3回リトライ後ログ記録」と、いずれも機械判定可能な形で明記されている。
- 「閾値は初期値70点とし、暫定値であることを仕様上明記する。運用開始後2週間のアラート発火実績(誤検知件数・見逃し件数)を記録し、閾値の妥当性を見直す」と記載されているが、「誤検知」「見逃し」の定義(判定手順・判定者・判定基準)は本文中に存在しない。
- 計測項目に「アラート発火回数・誤検知件数・見逃し件数(週次で振り返りメモに記録)」とあり、記録手段が「振り返りメモ」という手動記録である旨が明記されている。自動ログからの集計ではなく人手記録である前提が事実として記載されている。
- 「実際の放置時間の変化(アラート導入前後で比較)」が計測項目として挙げられているが、放置時間の起点・終点をどう記録するか(タイムスタンプの取得方法)についての記載はない。背景セクションの「過去1ヶ月のふりかえりで...平均放置時間が約90分だったことをログから確認した」という記述はあるが、これは過去実績の把握方法であり、導入後の継続測定方法ではない。
- 健康度スコアの算出式(差し戻し率60%・平均サイクル所要時間40%の加重平均)自体の計算正しさを検証する受入基準(単体テストケース)は、受入基準セクションに含まれていない。受入基準はアラート通知の挙動(投稿・本文・マスキング・リトライ)のみを対象としている。
- クールダウン(30分間の再通知抑制)についても、受入基準セクションに検証項目が存在しない。

## Risks

**[最重要] 「誤検知」「見逃し」の判定基準が未定義であり、2週間後の閾値見直しという仕様の核となる検証プロセスが実行不能または恣意的になる。** 「暫定値であることを仕様上明記する」「発火実績を記録し妥当性を見直す」という構造自体は良い設計だが、何をもって誤検知(スコアは閾値未満だが実際は健全だった)・見逃し(スコアは閾値以上だが実際は詰まっていた)と判定するかの操作的定義がなければ、2週間後の記録は評価者の主観に依存し、閾値見直しの判断が再現性・検証可能性を持たない。

- 放置時間(90分→15分の目標)の計測方法が未定義であり、機能導入の効果測定そのものが実行できない可能性がある。目標値が定量的に設定されている点は良いが、測定手段(起点・終点の記録方法)が仕様に含まれていない。

- 健康度スコア算出式(加重平均)自体の正しさを検証する受入基準がない。通知の周辺挙動(投稿・マスキング・リトライ)は網羅的にテスト可能だが、スコア計算ロジックにバグがあってもテストで検出されない可能性がある。

- クールダウン抑制(30分)の挙動を検証する受入基準が存在しない。仕様上の重要な副作用(連続発火の抑制)がテスト対象から漏れている。

## Recommendation

- 「誤検知」「見逃し」の判定手順を1〜2行で追記する(例:「アラート発火時点でループが実際に詰まっていたかを翌営業日に手動確認し記録する」など、簡易でよいので操作的定義を与える)。
- 放置時間の計測について、起点(詰まり発生とみなす条件)・終点(気づいたとみなす条件、例:担当者がSlackスレッドに反応した時刻)を1行で定義する。
- 受入基準に、クールダウン抑制の動作確認(同一原因で30分以内は再通知されないこと)を1項目追加する。
- スコア計算式の正しさを検証する単体テストケース(既知の入力に対する期待スコア値)を受入基準またはテスト計画に1件追加する。
