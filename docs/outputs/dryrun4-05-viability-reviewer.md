# viability-reviewer 生出力

## Assumptions

- $2/月という試算の内訳(トークン単価・入出力比率・Haikuの単価根拠)は成果物に記載がないため、計算過程は不明。
- 実装2日・テスト1日の見積もりが誰の工数(本人のみか、レビュー含むチーム工数か)を指すか記載がないため不明。
- 「今週末のボルトで着手」とあるが、ボルトの単位期間(1日か複数日か)が成果物に記載がなく、2日+1日の合計3日がそのボルト内に収まるか判断できない。
- 週次振り返りメモの記録・確認作業自体の工数(誰が・どれくらいの時間をかけるか)は記載がない。

## Findings

- 「Claude Haiku利用時、月間コストは概算で$2程度」と記載されており、発火頻度は「1日あたり平均3回」、ログサイズは「直近10サイクル分で平均2,000トークン」とされている。3回/日×30日×2,000トークン=約18万トークン/月という粗い規模感であり、Haikuの単価水準からして$2程度という試算は桁として不自然ではない。
- 「予算上限アラートは設定しない(この金額であれば費用対効果を都度確認する必要はないと判断)」と明示されており、判断根拠(金額の小ささ)が理由として述べられている。
- 機能概要2・4に「マスキング処理」が2箇所(要約前・投稿前の二重チェック)存在し、それぞれ実装・テスト対象になる。受入基準にも機密パターンのテストケースが明記されている。これは工数の一部として計上が必要な要素だが、コスト試算・スケジュール記述にはマスキング処理自体の工数配分の内訳が明記されていない。
- 受入基準に「3回まで指数バックオフでリトライ」「エラー記録」といったリトライ・エラーハンドリング実装が要求されており、これも2日の実装見積もりに含まれる前提と読めるが、内訳の記載はない。
- 計測項目に「アラート発火回数・誤検知件数・見逃し件数(週次で振り返りメモに記録)」とあり、継続的な運用コスト(人手による記録・確認)が発生する設計になっている。

## Risks

**[最重要] $2/月という試算の前提(発火頻度・ログサイズ)が実績ベースかどうか裏付けが薄い。** 「直近1ヶ月のループ詰まり実績から」1日3回としているが、この機能自体が稼働することで「詰まりの検知精度が上がる」「クールダウン30分で頻度が変わる」等、発火頻度が試算の前提から変動しうる。試算の変動幅(例:1日10回に増えた場合でも$10程度に収まるか)が示されておらず、桁が変わるほどの逸脱があるかは成果物からは判断できない。ただし予算上限アラートを設けない判断自体は、$2という金額の小ささを踏まえれば妥当な範囲。

- 週次振り返りメモへの記録という運用コストが継続的に発生するが、これがボルト外の「地味な保守作業」として続くかどうかは不明。3日の初期実装コストは小さいが、記録作業が形骸化・放置されるリスクは一般的なアラート機能によくあるパターン。
- 実装2日・テスト1日=3日の見積もりに対し、二重マスキング処理・リトライ機構・スキーマ検証という3つの独立した実装要素が含まれており、内訳なしの一括見積もりのため超過リスクの大小が判断できない。

## Recommendation

- $2/月試算に加えて、「発火頻度が想定の3〜5倍になった場合でも月$10未満に収まる」といった一言の感度確認を仕様に添えると、予算上限アラート不要という判断の説得力が上がる(実装は不要、記述の追加のみ)。
- 週次振り返りメモの記録が誰の作業として何分程度を想定しているか、一行で明記しておくと、継続コストの見積もりが後から検証しやすくなる。
