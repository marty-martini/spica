# verification-measurement-reviewer 生出力(3回目ドライラン)

## Assumptions

- 「アラートが正しく飛ぶこと」「通知の内容が分かりやすいこと」「ダッシュボードのテーマがダークになっていること」以外の受入基準は記載がないため、これが全量かどうかは不明。
- 閾値70点が何の指標(スコア種別・単位・算出式)に対するものかの定義が抜粋に記載がないため不明。
- 通知内容に含めるべき情報項目(スコア値・閾値・対象・タイムスタンプ等)の仕様が記載がないため不明。
- 「正しく飛ぶ」の判定条件(発火タイミング、対象条件、re-alert/重複抑制の扱い)が記載がないため不明。

## Findings

- 受入基準に「通知の内容が分かりやすいこと」とあるが、何をもって「分かりやすい」と判定するかの基準(必須記載項目、フォーマット、文言例など)が記載されていない。construction-loop の Checker が機械判定できる文言になっていない。
- 受入基準に「アラートが正しく飛ぶこと」とあるが、「正しく」の定義(発火条件・閾値との関係・タイミング・対象範囲)が本文中に明記されておらず、この一文単独では合否判定基準として不十分である。
- 受入基準に「ダッシュボードのテーマがダークになっていること」とあり、これは目視/スタイル確認で機械判定可能な粒度で書かれている。3基準の中では唯一、検証可能性が確保されている。
- 技術メモに「閾値(70点)は Slack 通知モジュール内の定数として直接埋め込む」とあるが、この閾値の設定根拠(なぜ70点か)や妥当性検証の記述がない。
- 技術メモの実装方針(定数埋め込み)は、閾値の調整可能性(運用中の変更)や、閾値変更が正しく反映されたかを事後検証する仕組み(設定値のログ出力、変更履歴等)についての言及がない。
- 受入基準・技術メモのいずれにも、アラート発火の成功率・誤検知率・通知到達率などを事後計測するための KPI や Telemetry/Logging の記載がない。

## Risks

**[最重要] 「通知の内容が分かりやすいこと」が定性的な表現のまま受入基準に採用されており、機械判定不可能である。** 誰が判定しても同じ結果になる客観的基準(必須項目のチェックリスト化、文言テンプレートとの一致確認など)が定義されない限り、construction-loop の Checker はこの基準を判定できず、人手によるレビューに依存し検証の再現性が失われる。

- 閾値70点の根拠が示されておらず、かつ定数埋め込みという実装方針により、閾値が妥当かどうかを事後的に検証する手段(A/Bテスト、閾値変更のログ、誤検知率のモニタリングなど)が仕様上存在しない。誤った閾値設定に気づく仕組みがないまま運用される懸念がある。
- 「アラートが正しく飛ぶこと」も抽象度が高く、発火条件・タイミング・対象範囲を明記した具体的な検証手順(例: スコアが69→71に変化した際にN秒以内に通知が送信される、など)がないため、この基準単体では自動テスト化できない。

## Recommendation

- 「通知の内容が分かりやすいこと」を、必須記載項目のチェックリスト(例: スコア値・閾値・対象名・発生時刻・アクションリンクを含むこと)に置き換え、機械判定可能な基準にする。
- 「アラートが正しく飛ぶこと」を、発火条件(閾値超過/以下遷移など)・許容遅延時間・対象範囲を明記した検証可能な文に分解する。
- 閾値70点の設定根拠を仕様に一言でも記載し、少なくとも「閾値超過によるアラート発火回数」「誤検知として報告された件数」など事後に閾値の妥当性を確認できる最小限のログ項目を計測対象に加える。
