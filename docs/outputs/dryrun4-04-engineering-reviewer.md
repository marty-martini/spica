# engineering-reviewer 生出力

## Assumptions

- 「fs.watch相当」がNode.js標準の`fs.watch`を指すのか、`chokidar`等のポーリング併用ライブラリを指すのか、成果物からは判別できないため不明として扱う。
- 「既存のヘルスチェック機構(spica本体が使用中のもの)」の具体的な監視間隔・検知内容(プロセス生存のみか、処理の進行性まで見るか)は成果物に記載がなく不明。
- pm2の再起動ポリシー(`autorestart`、`max_restarts`等の設定値)についての記載はなく不明。
- クールダウン状態(直前通知時刻)やリトライ状態をどこに永続化するか(メモリ/ファイル/DB)の記載はなく不明。
- ローカルログの出力先パス・ローテーション・保持期間についての記載はなく不明。
- 「60秒以内にSlack投稿成功」の起点が「state.json更新イベント検知時」か「スコア計算完了時」か明記がなく不明。

## Findings

- 「1. `.loop/state.json` の更新をファイル監視(fs.watch相当)で検知し...(ポーリングではなくイベント駆動)」と書かれているが、プロセスが再起動中・停止中に発生した`state.json`更新をどう扱うか(起動時の初期スキャンや取りこぼし救済)についての記述がない。
- 技術メモに「ファイル監視の実行主体はNode.jsの常駐プロセス(pm2管理)とし、プロセス死活監視は既存のヘルスチェック機構に相乗りする」とあるが、相乗り先のヘルスチェックが「プロセスが生きているか」のみを見るのか「監視が実際に機能しているか(イベントを取りこぼしていないか)」まで見るのかの区別がない。
- 受入基準に「Slack API呼び出し失敗時、3回まで指数バックオフでリトライし、最終失敗時はローカルログにエラー記録が残ること」とあるが、バックオフの初期待機時間・上限・ジッターの有無など具体的パラメータの記載がない。
- 「同一原因での再通知は、直前の通知から30分間はクールダウンとして抑制する」とあるが、このクールダウン状態を保持する場所(インメモリかファイルか)の記載がなく、pm2によるプロセス再起動時にクールダウン状態が消える(=再起動直後に再通知が発生しうる)かどうかが読み取れない。
- 受入基準「健康度スコアが閾値を下回ってから60秒以内にSlack投稿が成功すること」とあるが、コスト試算欄でHaikuによる要約処理(LLM呼び出し)が発火フローに含まれることが示唆されており、LLM応答時間・マスキング二重チェック・fs.watch検知遅延を合算した際に60秒制約に収まる根拠(レイテンシ試算)が本文にない。
- 「最終失敗時はローカルログにエラー記録が残ること」とあるが、そのログを誰がどの頻度で確認するか(アラート機能自体が落ちた際の二次検知手段)についての記述がない。

## Risks

**[最重要] プロセス停止・再起動中に発生した`.loop/state.json`更新がイベント駆動監視の設計上取りこぼされる可能性がある。** fs.watch系のイベント駆動監視は監視プロセスが起動している間のイベントしか捕捉できないため、pm2再起動(デプロイ・クラッシュ・ヘルスチェック起因の再起動含む)のタイミングでstate.json更新が起きると、そのサイクルのアラート判定自体が発火しないおそれがある。仕様には起動時の初期状態チェックやリカバリ処理の記載がなく、ポーリングを明示的に排除している分、この欠落がそのまま検知漏れリスクになる。

- クールダウン状態の永続化方法が未定義のため、pm2再起動のたびにクールダウンがリセットされ、再起動直後に同一原因の通知が連続する(スパム化)、または逆に永続化先の破損でクールダウンが解除されず正当なアラートが抑制され続ける、のどちらの障害モードもあり得る。
- 「既存のヘルスチェック機構への相乗り」が具体化されておらず、プロセスは生きているが監視ループ自体がフリーズ・例外で停止している(ゾンビ化)ケースを検知できない可能性がある。単純なプロセス死活監視だけでは、放置時間を15分に短縮するという目的に対して見逃しの温床になり得る。
- リトライの具体的パラメータ未定義のため、実装依存で待機時間が長くなり60秒受入基準に抵触する、または短すぎてSlack側のレート制限に抵触する余地が残る。

## Recommendation

- 起動時(pm2起動・再起動直後)に`.loop/state.json`を1回読み込んでスコア判定する初期スキャン処理を追加し、イベント取りこぼしの空白期間をなくす。
- クールダウン状態を`.loop/`配下などプロセス外のファイルに永続化し、再起動をまたいでも抑制ロジックが継続するようにする。
- 相乗り先ヘルスチェックが見ているものが「プロセス生存」か「監視ループの進行性」かを明記し、後者でない場合は最終イベント検知時刻をヘルスチェックが参照できる形で公開する(例:最終検知タイムスタンプをファイルまたはエンドポイントで露出)。
- 指数バックオフの初期値・上限・合計最大待機時間を明記し、60秒受入基準との整合性を数値で確認できるようにする。
