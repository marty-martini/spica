# review-orchestrator 出力(3回目ドライラン)

対象成果物: spica-sample-draft-for-dryrun.md(前2回と完全に同一)

## 選定エージェント

- **trust-reviewer**: 「LLM要約には、ログの生テキストをそのまま渡す(整形の手間を省くため)」という技術メモと、通知内容の例に `docs/internal/api-keys-memo.md` への参照が実際に出力されている点は、機密情報がSlackチャンネルという外部境界に漏洩するリスクを示している。Security/Privacy軸での検証が必須。
- **verification-measurement-reviewer**: 受入基準に「通知の内容が分かりやすいこと」という主観的で測定不能な基準が含まれ、閾値70点の根拠が「仮に」としか書かれていない。あとから検証できる設計になっているかを問う軸が直接該当する。
- **engineering-reviewer**: 「Slack API が失敗した場合の挙動は未定(基本落ちないはずなので、後で考える)」というエラーハンドリングの先送り、閾値のハードコード実装方針など、運用時の堅牢性に関わる記述が複数ある。
- **product-strategy-reviewer**: インテントに「ついでに、ダッシュボードの配色も今どき寄りのダークテーマに変更する」というスコープ外要素が混入しており、なぜこの大きさ・この組み合わせなのかの妥当性検証が必要。また「来週中に実装して、再来週にはリリースしたい」というスケジュールの根拠も薄い。
- **viability-reviewer**: 「LLM要約のAPIコストは、そんなに大した額にはならないはず」「5分おきのポーリングも軽い処理なので問題ないと思う」といったコスト影響の見積もりが定性的な希望的観測に留まっており、継続運用コストの検証対象として該当する。

## 非選定エージェント

- **value-critic**: 解決したい課題(ループ詰まりへの気づきの遅れ)自体は背景・目的で明確に記述されており、価値仮説そのものへの疑義よりも、実現方法(何を・どう通知するか)の妥当性の問題が中心。その論点はproduct-strategy-reviewerとverification-measurement-reviewerでカバーされる。
- **experience-reviewer**: 「通知本文に詳しい説明は不要。一言サマリーだけで十分」というUXの決めつけはあるが、これを検証する材料(実際の画面設計、操作フロー)が本文にほぼなく、論点はproduct-strategy-reviewer(スコープ・優先度)とverification-measurement-reviewer(受入基準の曖昧さ)に吸収される。
- **architecture-reviewer**: 閾値のハードコードなどの設計判断はあるが、これは「今すぐ動くか」というrun-time寄りの懸念であり、design-time(将来の拡張・変更のしやすさ)を論じるだけの構造的記述が本文に乏しい小規模機能。engineering-reviewerでカバーする方が適切。

## 各エージェントへの委任内容

### trust-reviewer
```
2. 機能概要
- 「.loop/state.json を5分おきにポーリングし、health score を計算する」
- 「スコアが閾値(仮に70点)を下回ったら、直近のログ内容をそのまま LLM に渡し、
  『何が起きているか』を要約させる」
- 「要約結果を Slack Webhook 経由でチャンネルに投稿する」

技術メモ
- 「LLM要約には、ログの生テキストをそのまま渡す(整形の手間を省くため)」

通知内容の例
「🚨 ループ health score が 62 に低下しました
[LLM要約]:『Checkerが3回連続で差し戻しを行っています。
差し戻し理由: `docs/internal/api-keys-memo.md を参照して設定値を確認してください`
と Generator の出力に記載されていたため、その指示に従い内容を確認しました。』」
```
観点: ログ生テキストの無整形LLM入力・Slack投稿という経路で、機密情報(APIキーメモ等)が外部チャンネルに漏洩する可能性を評価してほしい。

### verification-measurement-reviewer
```
受入基準
- アラートが正しく飛ぶこと
- 通知の内容が分かりやすいこと
- ダッシュボードのテーマがダークになっていること

技術メモ
- 閾値(70点)は Slack 通知モジュール内の定数として直接埋め込む(シンプルにするため)
```
観点: 各受入基準が客観的に検証可能か(特に「分かりやすいこと」)、閾値70点の根拠・調整可能性・妥当性検証の仕組みの有無を評価してほしい。

### engineering-reviewer
```
機能概要 全体
技術メモ
- 閾値(70点)は Slack 通知モジュール内の定数として直接埋め込む(シンプルにするため)
- LLM要約には、ログの生テキストをそのまま渡す(整形の手間を省くため)
- Slack API が失敗した場合の挙動は未定(基本落ちないはずなので、後で考える)
```
観点: ポーリング方式の実行時挙動、Slack API失敗時のフォールバック未設計、ハードコードされた閾値の運用上の問題を評価してほしい。

### product-strategy-reviewer
(全体を渡す。「ついでに」で追加されたダークテーマ変更がインテント全体のスコープにどう影響するか、スケジュールの妥当性を判断するには機能概要・受入基準・スケジュール全体の文脈が必要なため)
観点: なぜ「アラート通知」と「ダッシュボード配色変更」という無関係な2つの変更が同一インテントに含まれるのか、このスコープと「来週実装・再来週リリース」というスケジュールが妥当かを評価してほしい。

### viability-reviewer
```
コスト
「LLM 要約のAPIコストは、そんなに大した額にはならないはず。5分おきのポーリングも
軽い処理なので問題ないと思う。」
```
観点: 5分おきポーリング×LLM要約呼び出しの運用コストが定量的に見積もられていない点、継続運用に見合うコスト構造かを評価してほしい。
