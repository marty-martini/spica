# review-synthesizer 統合レポート(3回目ドライラン)

## サマリ(3行以内)
アラート通知機能とダークテーマ変更が単一インテントに束ねられている状態で、ログ生テキストをそのまま外部LLM・Slackに流す設計にPrompt Injection/機密情報漏洩の実例が既に現れている。受入基準は3項目中2つが機械判定不能、コストとエラー処理(Slack失敗時・閾値運用)も未定のまま「来週実装・再来週リリース」というスケジュールだけが確定している。

## 重大度別 所見

1. **ログ生テキストを無整形でLLM要約に渡し、そのままSlack投稿する設計により機密情報漏洩・Prompt Injectionの実例が機能概要自体に現れている。** 通知例に `docs/internal/api-keys-memo.md` への言及とそれに「従い確認しました」という記述が含まれ、ログ中の指示文をLLMが実行してしまう経路が既に存在する。要約→投稿間に人間レビュー・機密情報検知・redactionの仕組みが一切ない。 (出典: trust-reviewer)

2. **アラート通知機能とダッシュボード配色変更(ダークテーマ)という、目的も受益者も異なる2つの変更が同一インテントに束ねられている。** 「背景・目的」節はアラート機能の動機のみを説明し、配色変更の動機は「ついでに」の一言のみ。一方が完了してももう一方が未完了だとインテント全体の完了判定が曖昧になる。 (出典: product-strategy-reviewer)

3. **Slack API失敗時の挙動が「基本落ちないはずなので、後で考える」として未設計。** health score低下という最も通知すべき瞬間に通知がsilentlyに失われうる、監視システムのシングルポイント障害。 (出典: engineering-reviewer)

4. **受入基準3項目中2項目(「アラートが正しく飛ぶこと」「通知の内容が分かりやすいこと」)が機械判定不能な定性的表現。** 発火条件・タイミング・対象範囲、通知の必須記載項目が定義されておらず、construction-loopのCheckerが判定できず人手レビューに依存する。「ダッシュボードのテーマがダークになっていること」のみ検証可能な粒度。 (出典: verification-measurement-reviewer)

5. **LLM要約のAPIコストが「そんなに大した額にはならないはず」という感覚値のみで、定量試算が一切ない。** ログを切り詰めず「そのまま」渡す設計かつ閾値割れ継続時の呼び出し頻度抑制もないため、障害調査で頻発しうる悪化時にコストが跳ね上がる非対称なコスト構造が見落とされている。上限のないランニングコストという継続可否に直結するリスク。 (出典: viability-reviewer)

6. **閾値70点がコード内定数としてハードコードされ、設定根拠も記載がない。** 運用中の調整には毎回デプロイが必要になり、かつ妥当性を事後検証する手段が仕様上存在しない。 (出典: verification-measurement-reviewer, engineering-reviewer — 両者とも同一箇所を指摘、前者は「事後検証手段の欠如」、後者は「運用時の変更コスト」の観点)

7. **ポーリング処理自体の異常終了・ハングを検知する仕組みがなく、「監視システムが監視されていない」死角がある。** また同一障害継続時の通知重複抑制設計も不明で、5分おきの連投によるアラート疲れの懸念がある。 (出典: engineering-reviewer)

8. **「来週実装・再来週リリース」というスケジュールの根拠が本文に記載がない一方、技術メモにはSlack API失敗時の挙動という未決事項が明記されたままスコープ・スケジュールを固定しようとしている。** (出典: product-strategy-reviewer)

## ⚔️ 対立点

今回、各エージェントの指摘は方向性としては重複・補強関係にあり、明確な「結論の対立」は検出されなかった。ただし以下は対応の優先順位づけにおいて緊張関係を孕む可能性がある点として記録する。

- **スコープに対する立場の違い**: product-strategy-reviewer は「インテントを分割し、アラート機能側は未決事項をスコープ確定前に解消するか意図的にv0.1スコープ外にするか判断せよ」と、スコープ縮小・明確化の方向を提案。一方 engineering-reviewer は Slack失敗時のリトライ・ローカルログ出力を「後で考えるのではなく初期実装に含める」ことを提案しており、スコープを広げる(初期実装に追加する)方向。両者は「未決事項をどう扱うか」で、縮小(先送りを明示)と拡大(今すぐ実装)という異なる解決の方向性を示しており、どちらを採るかは裁定していない。

## 開発者への問い

1. アラート通知機能とダークテーマ変更は、インテントとして分割するか、それとも束ねたまま進めるか。束ねる場合、配色変更の背景・目的をどう補うか。
2. ログをLLM要約に渡す前段(サニタイズ・機密情報マスキング)およびSlack投稿前段(機密情報検知)を、来週のリリーススコープに含めるか、それとも明示的にv0.1スコープ外とするか。
3. Slack API失敗時のフォールバックとコスト上限(ログ切り詰め・クールダウン)を、初期実装に含めるか、後続タスクとして切り出すか。
