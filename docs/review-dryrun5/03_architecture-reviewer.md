# architecture-reviewer 生出力

## Assumptions

- マスキング処理(要約前・要約後の2回)が同一の実装(共通関数/モジュール)を指すのか、それぞれ別実装なのかが仕様に明記されていないため不明。
- `.loop/config.json`のスキーマ定義(閾値以外にどの項目を含むか、バージョニングの有無、ファイル不在時のデフォルト挙動)が記載されていないため不明。
- 既存のヘルスチェック機構(spica本体が使用中のもの)のインターフェース仕様(監視対象の登録方法、失敗時のコールバック契約など)が本ドラフトに記載がなく不明。
- 機密パターン(`internal/`パス、APIキー様文字列)の一覧がコード側でどう管理されるか(正規表現の定義場所)が記載されていないため不明。

## Findings

- 「技術メモ」に「閾値(70点)は`.loop/config.json`に外出しし、コード変更なしで調整可能にする」と明記されている一方、同じ「健康度スコアの算出方法」節にあるCheckerの重み(60%/40%)や「直近10サイクル分」という窓幅は外出し対象として記載されておらず、コード内定数と読める。
- 「同一原因での再通知は、直前の通知から30分間はクールダウンとして抑制する」というクールダウン値も、`.loop/config.json`への外出し対象として明記されていない。
- 機能概要2で「既知の機密パターン(`internal/`配下パスの言及、APIキー様文字列)をマスキング処理」と記載され、機能概要4で「要約結果に対しても同じマスキング処理を再度通す(二重チェック)」と記載されているが、「同じマスキング処理」が同一実装の再利用を指すのか、要約後専用の別ロジックなのかの区別がない。
- 機密パターンのリストは技術メモの外出し対象(閾値)には含まれておらず、パターン追加(新種の秘密情報検出)にはコード変更が必要と読める記述になっている。
- 技術メモに「プロセス死活監視は既存のヘルスチェック機構(spica本体が使用中のもの)に相乗りする」と記載されているが、当該機構のインターフェース(登録API、通知方式、失敗判定基準)についての記述はない。

## Risks

**[最重要] 「外出し」の基準が閾値のみに限定されており、同じ調整可能性を必要とする他のパラメータ(重み・窓幅・クールダウン時間)が設計判断として一貫していない。** 運用開始後2週間の実績を踏まえて閾値だけでなく重み付けや評価窓幅の調整が必要になった場合(仕様の「計測項目」にある誤検知/見逃し件数の振り返りは通常、閾値単体では解決しないケースが多い)、結局コード変更とデプロイが必要になり、「コード変更なし調整」という設計意図が部分的にしか達成されない。

- マスキング処理が要約前後で「同じ処理」であることが実装レベルで保証されていない(仕様上の言及のみ)。もし2箇所で別々に実装されると、片方だけパターン追加が漏れる形でドリフトし、二重チェックの意味が薄れる。マスキングパターン自体もコード内定数と読めるため、閾値のような外出しの恩恵を受けず、新しい機密パターンの追加のたびにコード変更・再デプロイが必要になる。
- 既存ヘルスチェック機構への「相乗り」は、当該機構のインターフェース変更(spica本体側の都合による改修)が本アラート機能の死活監視に無断で波及する結合を生む。相乗り契約(何を渡せば監視対象として認識されるか)が仕様に定義されていないため、将来的に監視対象を増やす・切り離す際の変更コストが見積もれない。

## Recommendation

- `.loop/config.json`に含める調整可能パラメータの範囲を明確化する(閾値だけでなく、重み・評価窓幅・クールダウン時間も対象に含めるか、含めない理由を仕様に一言追記する)。
- マスキング処理を単一の共有関数として実装する方針を仕様レベルで明記し、機密パターンのリストも`.loop/config.json`側(またはそれに準ずる外部定義ファイル)に置く方針を検討する。
- 既存ヘルスチェック機構との連携部分について、相乗り先が要求する最小限のインターフェース(登録方法・失敗判定基準)を仕様に1〜2行追記し、依存の境界を明示する。
