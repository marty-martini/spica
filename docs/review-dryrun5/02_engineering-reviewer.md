# engineering-reviewer 生出力

## Assumptions

- スコア計算処理そのものの所要時間(60秒予算のうちどれだけ消費するか)が成果物に記載がないため不明。
- 「ファイル監視」の実装詳細(fs.watchの既知の制限—rename検知漏れ、エディタのatomic writeによるinode差し替え未検知等)への対策記載がないため、対応有無は不明。
- pm2管理下のプロセスがクラッシュ後に再起動された際、直前のクールダウン状態(30分カウンタ)やLLM入力用の直近10サイクルログが永続化されているか、揮発するかの記載がないため不明。
- 「既存のヘルスチェック機構への相乗り」の具体的な検知間隔・アラート先が成果物に記載がないため不明。
- Slack API呼び出し自体のタイムアウト値が記載がないため、60秒予算内にリトライ3回(指数バックオフ)が収まる設計かどうか不明。

## Findings

- 機能概要1に「ファイル監視(fs.watch相当)で検知し」とあるが、fs.watchはプラットフォーム依存で二重発火や取りこぼしが知られる代表的なAPIである旨の言及・対策が本文中にない。
- 受入基準に「60秒以内にSlack投稿が成功すること」とあるが、これはハッピーパス(1回成功)の基準であり、指数バックオフで3回リトライするケースの合計所要時間が60秒制約とどう両立するかの記載がない。
- 技術メモに「プロセス死活監視は既存のヘルスチェック機構に相乗りする」とあるが、監視対象は「プロセスが生きているか」のみであり、fs.watchのイベント購読が内部的に停止・脱落した場合(プロセスは生存だが監視が機能していない状態)を検知する手段が記載されていない。
- 責務5「同一原因での再通知は、直前の通知から30分間はクールダウンとして抑制する」とあるが、このクールダウン状態の保持先(メモリ上かファイルか)の記載がなく、pm2によるプロセス再起動時に状態がリセットされるかどうかが不明。
- 受入基準「最終失敗時はローカルログにエラー記録が残ること」とあるが、ログの記録先・保存期間・監視者への到達経路(ログを能動的に見に行く仕組みか)についての記載がない。

## Risks

**[最重要] fs.watchのイベント取りこぼし・重複発火に対する対策が明記されておらず、「イベント駆動でスコアを毎回計算する」という機能概要1の前提自体が壊れるリスクがある。fs.watchはOS・ファイルシステムによってrenameイベントを見逃す、同一書き込みで複数回発火する等の既知の癖があり、これが未対策のまま実装されると「更新を検知できず健康度アラートが発報されない」という機能全体のサイレント障害に直結する。**

- Slack投稿の60秒受入基準と、3回指数バックオフリトライの所要時間の関係が未定義。バックオフ間隔次第では60秒を容易に超過し、受入基準の判定タイミング(初回試行基準か最終結果基準か)も曖昧なまま実装が進むリスクがある。
- ヘルスチェックの相乗り対象が「プロセスの生死」に留まる場合、fs.watchのリスナーが内部的に外れてもプロセス自体は生きているため異常が検知されず、アラートが無音のまま止まり続ける「静かな停止」のリスクがある。
- クールダウンやログ直近10サイクルの状態がメモリ保持だとすると、pm2の自動再起動(クラッシュ、デプロイ、OOM等)のたびに状態が失われ、意図しない再通知の連発、またはクールダウン適用漏れが発生しうる。

## Recommendation

- fs.watch単体ではなくchokidar等の実績あるラッパー採用、またはfs.watch使用時の既知の制限(取りこぼし・重複)への対応方針を技術メモに一行追記する。
- 「60秒以内」の起点(検知時刻か初回試行時刻か)と、リトライ込みで許容するかを受入基準に明記する。
- ヘルスチェック相乗りの粒度を「プロセス生死」から「直近N分以内にfs.watchイベントを1件も処理していない」ような軽量な生存確認(ハートビート)に一段引き上げることを検討する。
- クールダウン状態と直近10サイクルログの永続化先(ファイル/DB)をpm2再起動をまたいで保持するかどうかを技術メモに明記する。
