# verification-measurement-reviewer 生出力

## Assumptions

- 「Checker差し戻し率」「平均サイクル所要時間」の算出母集団(直近何件・何ループ分か)が抜粋内に記載されていないため不明
- 「誤検知件数・見逃し件数」の判定基準(何をもって誤検知/見逃しと判定するか、誰が判定するか)が記載されていないため不明
- 「実際の放置時間」の定義(アラート発火から何が起きるまでを指すか)が記載されていないため不明
- 通知本文スキーマの厳密な項目定義(型・フォーマット)は抜粋に含まれていないため不明

## Findings

- 受入基準に「健康度スコアが閾値を下回ってから60秒以内にSlack投稿が成功すること」とあり、時間制約と検証方法(テストケース)が明記されている。機械判定可能な形式になっている。
- 受入基準に「通知本文に、スコア値・閾値・差し戻し率・発生時刻の4項目が含まれること(必須フィールドのスキーマ検証で確認)」とあり、検証手段までセットで書かれている。
- 受入基準に「マスキング処理により、既知の機密パターン(...)が通知本文に含まれないこと」とあるが、「既知の機密パターン」はテストケースとして用意した2種(`internal/`パス参照・APIキー様文字列)のみであり、それ以外のパターンでの非検知は受入基準の対象外である旨が明記されていない。網羅性の限界が仕様上明示されていない。
- リトライ・失敗時のログ記録は「最終失敗時はローカルログにエラー記録が残ること」とあるが、ログの中身(何が記録されるべきか)のスキーマや、ログが残ったことをどう機械判定するか(ログフォーマット・検索可能な項目)が定義されていない。
- 計測項目に「アラート発火回数・誤検知件数・見逃し件数(週次で振り返りメモに記録)」とあるが、記録が「振り返りメモ」という手動運用に依存しており、システムログやTelemetryとしての自動計測手段が仕様に含まれていない。
- 「見逃し件数」を計測するには「本来アラートが発火すべきだった事象」を検知する仕組みが必要だが、その検知方法(閾値割れを見逃した事実をどう把握するか)が定義されていない。健康度スコア自体がアラート判定の唯一のトリガーである場合、見逃しの定義が循環しうる。
- 健康度スコアの閾値「初期値70点」について「暫定値であることを仕様上明記する」「運用開始後2週間のアラート発火実績を記録し、閾値の妥当性を見直す」とあるが、見直しの判断基準(誤検知率・見逃し率が何%を超えたら見直すか等の定量的トリガー)が記載されていない。「記録する」で止まっており「どう評価して意思決定するか」が未定義。
- 「実際の放置時間の変化(アラート導入前後で比較)」という計測項目があるが、「放置時間」の定義(スコアが閾値を下回ってから何がなされるまでの時間か)、および比較のベースライン取得方法(導入前データの有無)が記載されていない。
- (参考・担当外)Slackへのマスキング処理の実装方式・機密パターンの網羅的な定義そのものはSecurity観点の審査対象と思われるため、別軸での確認を推奨

## Risks

**[最重要] 「誤検知件数・見逃し件数」の判定基準が未定義であり、閾値見直しの土台となるKPIが計測不能になるおそれがある。特に「見逃し」は健康度スコアという単一シグナルの外側にある事象を検知する必要があるが、その検知手段が仕様化されていない。2週間後の閾値見直しが「感覚的な記録」に基づく判断になりかねない。**

- 受入基準の「マスキング処理」は用意された2種のテストパターンでのみ機械検証されるが、それ以外の機密情報パターンに対する非検知率は受入基準に含まれない。機密パターンの網羅性が担保されないまま「受入基準を満たした」と判定されるリスクがある。

- 計測項目が「週次で振り返りメモに記録」という手動プロセスに依存しており、自動Telemetryとしての実装が仕様に含まれていない。記録漏れ・記録者依存のばらつきが生じ、2週間後の閾値見直し判断の信頼性を損なうおそれがある。

- 閾値見直しの判断基準(定量的トリガー)が定義されていないため、「実績を記録する」だけで終わり、実際に見直すかどうかの意思決定プロセスが仕様上検証不能になっている。

- ローカルログの「エラー記録が残ること」は検証方法(ログの存在確認のみか、内容のスキーマ検証もするか)が曖昧であり、他の受入基準と比べて機械判定の厳密さに差がある。

## Recommendation

- 「誤検知」「見逃し」の判定定義を仕様に追記する(例: 誤検知=アラート発火後n時間以内に人間がスコア変化を「異常でない」と判定した件数、見逃し=事後にCheckerログから閾値割れが確認されたがアラートが発火しなかった件数、など)。
- 閾値見直しの定量トリガーを1行追加する(例: 「誤検知率が◯%を超える場合、または見逃しが1件以上発生した場合、閾値を見直す」)。
- マスキング受入基準に「テストケースで用意したパターンに限定した検証であり、未知パターンの非検知は保証しない」旨を明記するか、あるいは検証対象パターンのリストを拡充する。
- 「週次で振り返りメモに記録」の項目のうち、少なくとも「アラート発火回数」はシステムログから自動集計できる設計にし、手動記録は「誤検知/見逃しの判定」のみに限定する。
