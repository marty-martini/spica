# review-synthesizer 最終レポート(5回目ドライラン)

## サマリ(3行以内)
5軸とも「初期値・二重防御・外出し設計」までは揃っているが、各所で「宣言はあるが検証手段/永続化/裁定基準が未定義」という共通パターンが見られる。特にfs.watchの取りこぼし、機密パターンの網羅性未検証、誤検知/見逃しのKPI未定義の3点は互いに独立した最重要リスクとして重なっている。コスト・運用工数の異常検知手段の欠如も横断的な弱点。

## 重大度別 所見

1. **fs.watchのイベント取りこぼし・重複発火が未対策**。「イベント駆動で毎回スコア計算」という機能概要の前提自体が壊れ、更新を検知できず健康度アラートが発報されない「サイレント障害」に直結しうる。ヘルスチェックの相乗りも「プロセス生死」判定のみで、fs.watchのリスナー脱落(プロセスは生きているが監視が機能していない状態)を検知できない (出典: engineering-reviewer)。

2. **マスキングが既知2パターン(`internal/`パス・APIキー様文字列)限定のブロックリスト方式で、受入基準もその再現テストのみ**。未知の機密情報(氏名・トークン・個人データ等)の漏洩を検知する基準が存在しない。加えてLLM要約時のHallucinationにより、マスキング済み箇所を文脈から推測復元・言い換えして二重チェックをすり抜けるリスクがある (出典: trust-reviewer、verification-measurement-reviewerも受入基準の網羅性限界を同様に指摘)。

3. **「誤検知件数・見逃し件数」の判定基準が未定義でKPIが計測不能**。特に「見逃し」は健康度スコアという単一シグナルの外側の事象を検知する必要があるが手段が定義されておらず、循環しうる。閾値見直しの定量トリガーもなく、2週間後の見直しが「振り返りメモ」という手動記録に基づく感覚的判断になりかねない (出典: verification-measurement-reviewer)。

4. **状態永続化の設計が未定義**。クールダウン(30分)・直近10サイクルログの保持先(メモリかファイルか)が記載されておらず、pm2の自動再起動(クラッシュ・デプロイ・OOM)のたびに状態が失われ、再通知の連発やクールダウン適用漏れが起こりうる (出典: engineering-reviewer)。

5. **「コード変更なしで調整可能」という設計意図が閾値にしか適用されておらず一貫性がない**。Checkerの重み(60%/40%)・評価窓幅(10サイクル)・クールダウン値(30分)は外出し対象に含まれず、運用実績を踏まえた調整が結局コード変更・再デプロイを要する。機密パターンのリストも同様にコード内定数と読め、新規パターン追加のたびに再デプロイが必要 (出典: architecture-reviewer)。

6. **発火頻度が想定を超えた場合のコスト異常を検知する手段がない**。予算上限アラートを明示的に設定しない方針だが、月間$2試算は平均値のみに基づき、ピーク時・分散データの裏付けがない (出典: viability-reviewer)。

他13件の指摘あり(Prompt Injection対策の具体性不足、Slack Webhook管理・投稿先権限、60秒基準とリトライ時間の整合性、マスキング二重チェックの実装同一性保証、既存ヘルスチェック機構との結合契約未定義、ログ記録のスキーマ未定義、運用保守工数の未計上、優先順位の定量的機会費用比較の欠如 等)。詳細は各エージェントの出力を参照。

## ⚔️ 対立点
今回は明確な結論の対立は見られない。ただし焦点の置き方に差がある。

- architecture-reviewerは「外出し設計の一貫性」という構造的観点から重み・窓幅・クールダウンの外出し漏れを問題視する一方、engineering-reviewerは同じクールダウン値について「外出しの有無」ではなく「永続化(pm2再起動での揮発)」を問題視しており、同じパラメータに対して異なる軸から異なる改修(config化 vs 永続化)を示唆している。両方が必要な可能性があるが、どちらを優先すべきかは仕様に明記されていない。

## 開発者への問い
1. マスキングの受入基準を「既知2パターンの再現テストのみ」で確定としてリリースするか、それとも未知パターンへの対応(検証拡充または限界の明記)を先行させるか。
2. fs.watchの取りこぼし対策(chokidar採用等)とクールダウン/ログの永続化設計は、v0.2の受入条件に含めるか、次イテレーションに回すか。
3. 「誤検知/見逃し」の判定定義と閾値見直しの定量トリガーを仕様に明記してからリリースするか、2週間の運用実績を先に集めてから定義するか。
