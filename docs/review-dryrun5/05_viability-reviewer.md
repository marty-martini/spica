# viability-reviewer 生出力

## Assumptions

- 「直近10サイクル分で平均2,000トークン」の内訳(入力/出力トークンの比率、プロンプト内の固定コンテキスト量)が記載されていないため、Haiku単価に対する概算根拠を検証できない。
- 「直近1ヶ月のループ詰まり実績」の実測データ(最大値・分散)が記載されておらず、平均3回/日以外の記述がないため、ピーク時発火頻度は不明。
- 実装2日・テスト1日以降の運用工数(アラート精度のチューニング、誤検知対応、閾値見直し)についての記載がなく、ランニングの保守工数は不明。
- 通知チャネル(Slack/メール等)の記載がなく、通知配信自体のインフラ費・API費が試算に含まれているか不明。
- 「今月中に着手したい機能」の中での優先順位付けの根拠(期待効果の比較値)が記載されていないため、機会費用の妥当性を判断する基準が不明。

## Findings

- 「Claude Haiku利用時、月間コストは概算で$2程度」と、平均値(2,000トークン×平均3回/日)のみに基づいて算出されている。ピーク時やループ詰まりが多発する月の試算は記載されていない。
- 「予算上限アラートは設定しない(この金額であれば費用対効果を都度確認する必要はないと判断)」と明記されており、コスト超過を検知する仕組みが存在しない設計になっている。
- スケジュールは「実装2日・テスト1日」のみが記載されており、リリース後の監視・チューニングにかかる工数の見積もりが存在しない。
- 優先度は「spicaダッシュボードの他改善より後、蒼羽の作業より前」と相対順位のみで示され、絞り込みの定量的な判断基準(期待効果や代替手段との比較)は記載されていない。

## Risks

**[最重要] 発火頻度が平均値の想定を超えた場合のコスト増加を検知する手段がない。** 予算上限アラートを明示的に設定しない方針であるため、ループ詰まりが想定以上に多発する事象(たとえば実装バグによる誤検知の連鎖や、対象サイクル数の増加)が起きても、コストの異常増加に気づく仕組みが存在しない。「$2程度だから確認不要」という判断自体は、実測データの分散が不明な状態(Assumptions参照)で下されており、根拠が薄い。

- 実装3日分の工数は見積もられているが、リリース後の継続コスト(通知の誤検知チューニング、閾値の見直し、アラート疲れへの対応)が計上されていない。これらは一過性の実装コストと異なり、機能を「続ける」限り発生し続ける保守負荷であり、v0.2ドラフトの割に合うか判断において欠落している。
- 優先度判断が定性的な順位付け(「〜より後」「〜より前」)にとどまり、この3日間を他の作業に充てた場合との比較(機会費用)が定量化されていない。中位優先度という位置づけの妥当性を第三者が検証できない。

## Recommendation

- 予算上限アラートを設定しないと決める前に、直近1ヶ月のループ詰まり発生回数の最大値・分散を確認し、ピーク時の月間コストを併記する(平均$2に加え、最悪ケースの概算を1行追加するだけで足りる)。
- コスト自体を都度確認しない方針を維持するとしても、発火回数が「平均3回/日」から大きく逸脱した場合(例:1日10回超)にのみ検知する簡易な閾値監視を設定する。予算アラートではなく頻度アラートとして最小工数で追加できる。
- リリース後1〜2週間の「様子見」期間と、その間の想定工数(チューニング作業時間)を見積もりに明記する。
