# spica ビジョン文書

*この文書は AI-DLC ワークフローの Requirements Analysis への一次入力である。*

## spica とは

AIループ(AI-DLCのInception/Constructionサイクル)の実行を、AIの自己申告ではなく
**観測事実**として見えるようにするツール。開発者(個人)がループエンジニアリングを
練習・上達するための「感覚器官」である。

## 解決したい課題

現状、`/aidlc` やスキル・サブエージェントを使った開発では、
「スキルやエージェントが想定通りに呼び出されたか」「ちゃんと動いているか」を
確認する手段がなく、AIの自己申告を信じるしかない。
実行の中間状態を見たいときは、生ログを手作業でファイル保存して確認するしかない。

観測できないものは練習できない。spica はこの観測手段を提供する。

## 利用者

開発者本人のみ(ソロ開発)。判断基準は常に
「これを見て、次のループで自分の行動が変わるか?」。変わらない情報は表示しない。

## プロダクトの3層構造(ロードマップ)

1. **Trace(v1・今回のスコープ)**: 本当に何が起きたかの記録と閲覧
   - 実行イベント(スキル発火・エージェント呼び出し・ゲート承認・差し戻し)の記録
   - 各イベントのタイムスタンプと区間所要時間(承認待ち時間含む)
   - 閲覧手段は CLI で十分(タイムライン表示)。リッチな UI は後続フェーズ
2. **Conformance(将来)**: 実行が設計(想定したエージェント構成・ゲート通過)と
   一致していたかの照合
3. **Health(将来・最終ゴール)**: 差し戻し率・所要時間などの集計とダッシュボード表示

最終ゴールはダッシュボードだが、データの依存関係に従い Trace → Conformance → Health
の順に建設する。

## v1 の技術方針

- **データソースは公式 AI-DLC が生成する `aidlc-docs/`(特に audit.md)のパースとする**
  - 非侵襲的で、公式のバージョンアップに追従しやすい
  - Claude Code hooks による自前計測(より高粒度)は、audit.md の粒度不足を
    実際に3回感じてから検討する(拡張候補として設計上の考慮のみしておく)
- 言語は TypeScript
- 開発者の作業単位は細切れ時間(平日夜1h・週末2h)。1つの作業単位はこれに収まる
  粒度に分割する

## v1 のスコープ外(明記)

- グラフ・スコア・ダッシュボード UI(Health 層で扱う)
- 設計との照合機能(Conformance 層で扱う)
- リアルタイム監視・アラート通知
- 複数ユーザー対応・認証
