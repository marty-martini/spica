# ビジネスルール — unit: trace-cli

## BR-1: 非侵襲(読み取り専用)
`aidlc-docs/` 配下への書き込みは一切行わない。spica が書いてよいのは `.spica/` 配下のみ。

## BR-2: 記録を落とさない
タイムスタンプを持つエントリは必ずタイムラインに現れる。分類できなくても `unknown` として表示する(捨てるのはタイムスタンプ欠落等の構造不良のみで、その場合も件数と理由を報告する)。表示時は `unknownReason`(なぜどの分類キーワードにも一致しなかったか)を併記し、開発者が audit.md の書き方を見直す判断材料にする(NFR-2 の「なぜ計測・分類できないか」まで含めた正直な表示)。

## BR-3: 推定と事実を混同しない
- 承認待ち時間は、gate-prompt とユーザー入力を持つ応答イベントの両方が特定できた場合のみ数値表示する(1 対 1 対応。同一の応答イベントを複数の gate-prompt の待ち時間計算に使い回さない。ロジック 6.2 参照)
- 特定できない場合は「計測不能」と `unmeasurableReason`(理由: 応答イベントが見つからない)を表示する。近似値・推定値を出さない
- 区間所要時間(durationFromPrev)は事実(タイムスタンプ差)なので常に表示してよい。ただし「承認待ち時間」とは区別して表示する

## BR-4: 冪等性
同一の audit.md に対して CLI を何度実行しても、`.spica/events.jsonl` の内容は変化しない(追記 0 件・重複 n 件と報告される)。ただし ID は `timestamp + heading` のみに基づくため、本文編集による静かな不整合(4章の既知の限界)はこの冪等性の範囲外であり、検知対象としない。

## BR-5: ソート順の安定性
表示順は timestamp 昇順、同時刻はファイル内出現順(order)。実行のたびに順序が変わってはならない。

## BR-6: 表示タイムゾーン
永続化は記録どおり(ISO 8601 / UTC)。表示はローカルタイムゾーンに変換する。

## BR-7: エラー時の振る舞い
| 状況 | 振る舞い |
|---|---|
| audit.md が存在しない | 明確なメッセージ(対象パスを含む)+終了コード 1 |
| audit.md が空・エントリ 0 件 | 「イベントなし」と表示して正常終了(コード 0) |
| パース不能エントリあり | スキップして続行。件数を**先頭**(ダイジェスト)で報告し、見出し+理由の詳細は本体の後に表示する |
| events.jsonl に壊れた行 | 読み飛ばして続行。件数を報告 |
| .spica/ が存在しない | 自動作成する |

## BR-8: セッション概念を導入しない(v1)
サマリーはステージ単位の集計のみ。セッション分割(時間ギャップ・暦日)は行わない(Q3 = C)。

## 受入基準トレース

| 受入基準(requirements.md) | 担保するルール/ロジック |
|---|---|
| 1. タイムライン+サマリー表示 | ロジック 6.1 / 6.3、CLI フロー |
| 2. 承認待ちの計測可能/計測不能表示 | BR-3、ロジック 6.2(応答の 1 対 1 対応含む) |
| 3. 冪等な JSONL 蓄積 | BR-4、ロジック 4 / 5 |
| 4. パース不能でも正常終了+件数報告 | BR-2、BR-7 |
| 5. テスト・型チェックのパス | Code Generation / Build and Test で担保。最低限含めるテストケース: (a) BR-3 の measurable 1件+unmeasurable 1件を含むフィクスチャ、(b) 連続する複数 gate-prompt に対する応答の 1 対 1 対応、(c) 同一 audit.md を 2 回実行して events.jsonl の diff が 0 件になる BR-4 冪等性テスト |

## 既知の限界(v1 では対応しない)
- audit.md の本文(User Input / AI Response / Context)だけが編集され、見出し・タイムスタンプが不変の場合、EventStore はこれを検知しない(BR-4 参照)。contentHash 等による変更検知は過剰設計と判断し、v1 では実装しない
- `TraceEvent.source` に将来ソース(例 "hooks")が追加された際の ID 名前空間の衝突回避、および `order` フィールドのマルチソース間でのソート安定性は、実際にソースを追加する時点で設計する(ロジック 4 参照)
