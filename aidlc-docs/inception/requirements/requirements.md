# spica Trace 層 v1 — 要件定義

## インテント分析サマリー

- **ユーザーリクエスト**: spica の Trace 層 v1(aidlc-docs のパースと CLI タイムライン表示)を作りたい
- **リクエスト種別**: New Project(グリーンフィールド)
- **スコープ見積**: Multiple Components(パーサー / イベントストア / CLI 表示)
- **複雑度見積**: Moderate
- **一次入力**: `docs/spica-vision.md`、および `requirement-verification-questions.md` への回答
- **なぜ今か**: ユーザー本人が本セッションで着手を明示的に要求。3層ロードマップ(Trace → Conformance → Health)はデータ依存関係上 Trace が起点であり、着手可能になった時点が着手すべき時点である

## プロダクトコンテキスト

spica は AI ループ(AI-DLC の Inception/Construction サイクル)の実行を、AI の自己申告ではなく**観測事実**として見えるようにするツール。利用者は開発者本人のみ(ソロ開発)。表示する情報の判断基準は常に「これを見て、次のループで自分の行動が変わるか?」。

v1 は 3 層構造(Trace → Conformance → Health)の第 1 層のみを対象とする。まずは本リポジトリ自身のドッグフーディングから始める(別リポジトリの集計は v2 以降)。

## 機能要件

### FR-1: audit.md のパース
- `aidlc-docs/audit.md` を入力とし、エントリ(見出し・Timestamp・User Input・AI Response・Context)を構造化イベントに変換する。
- 抽出対象イベントは **audit.md から確実に取れるものだけ**とする(Q1: A):
  - ステージ開始/完了
  - 承認ゲート(承認/差し戻し)
  - ユーザー入力イベント
- 上記のうち audit.md の記録粒度から確実に判定できないものは、**無理に推定せず MVP では捨ててよい**(Q1 回答による)。スキル発火・エージェント呼び出しのテキスト解析による推定抽出は行わない。
- パースできないエントリはエラーで停止せず、スキップして件数を報告する(壊れた記録があっても全体は動く)。

### FR-2: 対象パスの指定
- 対象の `aidlc-docs/` ディレクトリを CLI 引数で指定できる(Q3: B)。
- 省略時はカレントディレクトリ直下の `aidlc-docs/` を使う。

### FR-3: タイムライン表示(CLI)
- イベントを時系列に、タイムスタンプ・イベント種別・ステージ名とともに一覧表示する。
- 各区間の所要時間を表示する。承認待ち時間は、承認プロンプト提示と応答が別エントリとして記録されている場合のみ計測し、計測できない場合は「計測不能」と明示する(Q2: A)。推定値を事実のように表示しない。

### FR-4: 所要時間サマリー
- タイムラインに加えて、セッション/ステージごとの所要時間サマリーを表示する(Q5: C)。
- グラフ・スコアリングは行わない(Health 層のスコープ)。

### FR-5: パース結果の永続化
- パース結果の正規化イベントを JSON として永続化する(Q6: B)。将来の Conformance/Health 層の入力となる蓄積データを兼ねる。
- **提案(承認レビュー対象)**: 保存先はワークスペース直下の `.spica/events.jsonl`(1 行 1 イベントの追記型 JSONL)。再パース時は同一イベントを重複登録しない(冪等)。audit.md 自体には一切書き込まない(非侵襲の原則)。

### FR-6: 実行形態
- 本リポジトリ内の npm scripts / `npx tsx` で実行する(Q4: A)。ビルド・配布・bin エントリは v1 では作らない。

## 非機能要件

- **NFR-1 (非侵襲性)**: データソースは公式 AI-DLC が生成する `aidlc-docs/` の読み取りのみ。既存ファイルを変更しない。公式フォーマットのバージョンアップに追従しやすいよう、パーサーは独立モジュールとする。
- **NFR-2 (正直な表示)**: 計測できないものは「計測不能」と表示し、推定と事実を混同させない。
- **NFR-3 (作業粒度)**: 開発の作業単位は細切れ時間(平日夜 1h・週末 2h)に収まる粒度に分割する。
- **NFR-4 (技術スタック)**: TypeScript。テストは Vitest、実行は tsx、型チェックは tsc の最小構成(Q7: A)。ESLint/Prettier の整備は v1 スコープ外。
- **NFR-5 (拡張余地)**: Claude Code hooks による高粒度計測は実装しないが、イベントスキーマはソース種別(`source: "audit-md"` 等)を持たせ、将来のソース追加を妨げない。

## スコープ外(明記)

- グラフ・スコア・ダッシュボード UI(Health 層)
- 設計との照合機能(Conformance 層)
- リアルタイム監視・アラート通知
- 複数ユーザー対応・認証
- 別リポジトリの集計(v2 以降)
- npm パッケージとしての配布
- スキル発火・エージェント呼び出しのテキスト解析による推定抽出

## 拡張ルール設定

| Extension | Enabled | Rationale |
|---|---|---|
| Security Baseline | No | ローカル完結・読み取り専用 CLI。外部通信・認証・秘密情報の取り扱いがなく、攻撃対象領域が実質存在しないため |
| Property-Based Testing | No | 対象ロジックはテキストパースと時間計算が中心で、複雑な数理的不変条件を持たない。Vitest の具体例ベーステストで十分な網羅性が得られるため |
| Resiliency Baseline | No | 単一ユーザー・ローカル実行の CLI で、高可用性・障害復旧の要求がそもそも存在しないため |

## 受入基準(v1 完了の定義)

1. 本リポジトリの `aidlc-docs/audit.md` を対象に CLI を実行すると、タイムラインと所要時間サマリーが表示される。
2. 承認待ち時間は計測可能な区間のみ数値表示され、それ以外は「計測不能」と表示される。
3. 実行後、`.spica/events.jsonl` に正規化イベントが蓄積され、再実行しても重複しない。
4. パース不能なエントリがあっても CLI は正常終了し、スキップ件数が報告される。
5. `npm test`(Vitest)と型チェックが通る。
